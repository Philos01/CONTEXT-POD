我们将“想法二”的自动识别优势与“想法三”的稳健闲时学习策略进行深度融合。

这款 **“伴聊悬浮舱”** 的核心进化逻辑是：**“像影子一样感知，在静默中进化”**。它不仅能通过本地 OCR 瞬间认出聊天对象，还能在你不使用电脑的空隙，自动复盘积攒的记录，精准“夺舍”对方或你的聊天风格。

以下是完整、详细的实施策略。

---

### 一、 核心想法逻辑：为什么这么做？

1.  **隐私防线 (Privacy First)**：微信不提供导出接口，云端 OCR 存在隐私风险。我们通过**本地 RapidOCR** 直接在内存中完成“截图 -> 文字”的转换，数据不离端。
2.  **零摩擦体验 (Zero-Friction)**：用户不需要手动选择“正在和王总聊天”。Agent 启动时自动定位微信窗口标题或气泡上方文字，实现**“即唤醒，即识别，即适配”**。
3.  **异步进化 (Asynchronous Evolution)**：大模型分析风格（Persona Extraction）是耗时且昂贵的（消耗 Token）。我们不放在生成回复的同步流程中，而是利用**闲时（Idle Time）**批量处理，保证回复时的“秒出”。
4.  **原子安全性 (Atomic Safety)**：在微信高频聊天的背景下，缓冲区是动态增长的。我们通过 **“UUID 锁定 -> 批量消化 -> 精准删除”** 的策略，确保在 AI 思考风格的几秒钟内，新产生的聊天记录不会被误删。

---

### 二、 技术实施方案

#### 1. 环境感知层：本地 RapidOCR 与自动路由
**目标**：按下快捷键后，0.5秒内识别出聊天对象并定位上下文。

*   **技术实现**：
    *   集成 `rapidocr-node-prebuilt` 到 Tauri 后端。
    *   **识别算法**：
        1.  **标题识别**：OCR 扫描屏幕顶部区域。微信窗口顶部的文本通常即为 `ContactName`。
        2.  **气泡对齐**：通过 OCR 返回的 `box` 坐标，区分左侧（对方）和右侧（我）的文字，存入不同的风格缓冲区。
*   **代码逻辑**：
    ```javascript
    const result = await rapidocr.execute(screenshotBuffer);
    // 寻找坐标 y < 50 的文本块，作为自动匹配的联系人 ID
    const contactName = result.find(item => item.box[1] < 50)?.text || "未知联系人";
    ```

#### 2. 数据持久层：基于 UUID 的 LanceDB 缓冲区
**目标**：安全、结构化地存储原始语料。

*   **表结构设计**：
    *   表名：`raw_chat_buffer`
    *   字段：`id (UUID)`, `contact_name (String)`, `content (Text)`, `role (Enum: User/Partner)`, `timestamp (Int)`。
*   **动作**：
    *   每次用户唤醒 Agent 并抓取成功后，立刻调用 `pushToBuffer()`。

#### 3. 调度层：双重阈值闲时触发器 (Idle Detector)
**目标**：精准捕捉电脑“空窗期”。

*   **触发条件**：
    *   **条件 A**：`raw_chat_buffer` 中单一联系人的记录数 $\ge 50$ 条。
    *   **条件 B**：用户窗口处于 `blur`（失焦/最小化）状态超过 30 秒，**且** CPU 占用率低于 15%。
*   **实现**：
    ```typescript
    // 监听窗口失去焦点
    window.addEventListener('blur', () => {
      idleTimer = setTimeout(async () => {
        const count = await db.table('raw_chat_buffer').count();
        if (count >= 50) triggerPersonaUpdate();
      }, 30000);
    });
    ```

#### 4. 执行层：UpdatePersonaNode (核心逻辑)
**目标**：利用 LangGraph 节点完成风格提炼，并保持数据一致性。

*   **步骤 1：快照锁定 (Snapshot Locking)**
    *   从 `raw_chat_buffer` 中提取最早的 50 条记录。
    *   **关键点**：记录这 50 条数据的 `UUID` 列表，称为 `Target_IDs`。
*   **步骤 2：风格建模 (Style Modeling)**
    *   调用大模型（推荐 DeepSeek V3，性价比极高）。
    *   **Prompt 策略**：不让 AI 总结事件，而是让它化身语言学家，提取：*语气（高冷/热情）、断句（长句/碎片）、标点偏好（~/. /无）、口癖、回复延迟特征等*。
*   **步骤 3：增量向量化 (Incremental Vectoring)**
    *   将生成的 `Persona_Summary` 存储到 LanceDB 的 `persona_profiles` 表中。
    *   如果该联系人已有画像，则进行“合并更新”：`new_persona = LLM(old_persona + new_50_logs)`。
*   **步骤 4：原子清理 (Atomic Clean-up)**
    *   **操作**：`DELETE FROM raw_chat_buffer WHERE id IN (Target_IDs)`。
    *   **逻辑**：只删除参与了本次计算的 50 条。如果在 LLM 思考期间产生的新记录（第 51、52 条），其 UUID 不在列表中，会被安全保留到下一批处理。

### 三、 完整架构流程图

1.  **捕捉阶段**：
    *   [用户] 按下 F9 -> [Tauri] 截屏 -> [RapidOCR] 识别文字与联系人 -> [前端] 弹出回复建议。
    *   同时，[后台] `INSERT INTO raw_chat_buffer` (异步执行，不卡顿界面)。

2.  **判定阶段**：
    *   [后台周期任务] 检查缓冲区。
    *   发现“王总”语料满 50 条 + 用户去喝咖啡了（Idle）-> 启动 `UpdatePersonaNode`。

3.  **学习阶段**：
    *   [UpdatePersonaNode] 锁定前 50 条 UUID -> 发送给云端 LLM。
    *   [LLM] 返回“王总风格画像 V2.0”。
    *   [LanceDB] 更新 `persona_profiles` 表。
    *   [LanceDB] 根据 UUID 列表精准删除 `raw_chat_buffer` 中的旧数据。

---

### 四、 这种方案的优势（为什么这是目前的最优解）

1.  **极低成本的“自进化”**：市面上的 Agent 大多是静态的，而你的 Agent 会随着用户聊天的深入越来越像真实的人。每 50 条一次的更新频率，既保证了风格的及时跟进，又避免了频繁调用 API 带来的高额成本。
2.  **工程级的鲁棒性**：通过 **UUID 锁定删除**，彻底解决了桌面软件中常见的“数据竞争”问题。无论用户聊得有多快，系统都不会丢失任何一条语料。
3.  **完美的隐私闭环**：OCR 是本地的，数据库是本地的，发给云端的只有经过处理的文本片段，且最终产出的“风格画像”也保存在本地。这极大地缓解了国内用户对“AI 监控聊天记录”的恐慌。

**实施建议：**
由于微信界面版本更新可能导致坐标变化，建议在 OCR 层增加一个**“动态特征锚点”**。即：先寻找微信特有的绿色图标或搜索框位置，以此为原点计算相对坐标，来准确定位联系人姓名和气泡文字。
